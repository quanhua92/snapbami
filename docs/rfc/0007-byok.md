# RFC-0007: Bring Your Own Key (BYOK)

> **Status:** Draft · **Active (Track T prerequisite)** — first needed when a tool calls a model
> **Author:** Quan
> **Created:** 2026-07-12
> **Depends on:** [RFC-0001](./0001-core-platform.md), [RFC-0003](./0003-pluggable-tools.md)
> **Resolves:** [DECISIONS.md D2](../DECISIONS.md)
> **Absorbs:** the deferred LangGraph model-gateway idea

---

## 1. Thesis

LLM cost should not be the platform's problem when a user wants more than the free
metered cap. BYOK lets a user store **their own model API key**; tools and (if ever
revived) the deferred pipeline prefer that key, so model calls are billed to the
user and platform compute cost stays at $0 for that path.

One line: **user stores a provider key; tools that need a model prefer it and lift the cap.**

BYOK is **not required** for the core publish loop. It first matters at
[Track T](../TODO.md) (tools that call models) and becomes important for paid tiers.

---

## 2. Data model

Uses the existing `byok_keys` table (already in `002_credentials.sql`):

| Column | Role |
|---|---|
| `user_id` | Owner |
| `provider` | e.g. `openai`, `anthropic`, `google`, `openrouter` |
| `label` | User-facing name (e.g. "Personal OpenAI") |
| `key_hash` / `key_encrypted` | Encrypted secret at rest (see §5) |
| `key_prefix` | Display-only prefix |
| `is_active` | Soft disable |
| `created_at` / `last_used_at` | Audit |

Invariant: BYOK is **per-user**, never per-workspace.

---

## 3. Key selection at call time

When a tool (or future pipeline node) needs a model:

```text
need a model for {provider, capability}
  │
  ▼
does the user have an active byok_keys row for that provider?
  ├─ yes  → use user key; do NOT count against platform-metered cap
  ▼ no
use platform key (system_credentials); count against per-user free-tier cap
  ├─ over cap → 402 / 429 (tier-dependent), prompt to add BYOK or upgrade
```

A tool never invents its own key handling — it asks a platform service for
"resolved credential for (user, provider, capability)".

---

## 4. REST surface

Auth: session JWT (web) or API token.

| Method | Path | Purpose |
|---|---|---|
| `POST` | `/api/credentials/byok` | Add a key (returns nothing sensitive; stores encrypted) |
| `GET` | `/api/credentials/byok` | List prefixes, labels, provider, last_used |
| `DELETE` | `/api/credentials/byok/{id}` | Revoke (`is_active = false` + wipe ciphertext) |

Full key material is shown **only at creation** (optional), and even then the
recommended flow is paste-once and never display.

---

## 5. Security

- **Encrypt at rest** with an app-level envelope key (KMS / age / libsodium
  secretbox). Store ciphertext + nonce, never plaintext. `key_prefix` is the only
  thing readable without unsealing.
- **Never log** key material — log `key_prefix` + provider.
- **Never return** key material from any GET endpoint.
- Decrypt only inside the narrow call path, in memory, for the duration of one
  model request.
- Rotate the envelope key on a schedule; re-encrypt on rotation.
- Revoke = wipe ciphertext + set `is_active = false` (not just a soft flag).

---

## 6. Tier interaction

| Tier | Platform key | BYOK |
|---|---|---|
| Registered free | Metered, hard cap | Optional — lifts model-call cap for BYOK-covered providers |
| Registered + BYOK | Fallback only | Preferred |
| Paid | Included up to plan | Optional / higher caps |

BYOK does **not** grant storage or rate-limit exemptions by itself — only model-call
quota. Storage tiers stay gated by `users.tier`.

---

## 7. Tool contract

In `ToolSpec` ([RFC-0003](./0003-pluggable-tools.md)):

```text
needs_model: { provider, capability } | null
```

- If `null`, the tool is deterministic and never touches credentials.
- If set, the runner resolves the credential per §3 before invoking the handler.
- Handlers receive a resolved `model_client`, never raw keys.

---

## 8. Relationship to the deferred pipeline

The old LangGraph model-gateway design is **absorbed here**, not re-built:

- The gateway concept = the "resolved credential" service in §3.
- Per-node model selection in a multi-agent graph would reuse the same resolver.
- If [RFC-0004](./0004-deferred-langgraph-pipeline.md) is ever revived, it calls
  into this RFC rather than owning key management.

So a future multi-agent pipeline adds **no new credential surface** — it reuses BYOK.

---

## 9. Non-goals

- **BYOK for storage** (S3 credentials) — not supported; storage is always
  platform-managed.
- **Key brokering / sharing between users** — that is teams ([Future](../FUTURE.md)).
- **Model routing optimization** (cost/latency fallback across providers) — future
  enhancement, not v1.

---

## 10. Open questions

1. Envelope-key store (env var vs KMS vs sealed file) for single-instance deploy.
2. Whether to support multiple keys per provider per user (e.g. rotation overlap).
3. UI for cap/usage visibility so users know when BYOK is worth adding.
